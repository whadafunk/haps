# Authentication & Identity Model

## Overview

There are four distinct identity tiers. They are not a hierarchy — they are
parallel lanes, each with its own auth mechanism and trust level.

```
Tier 1: Admin / Organizer  — email + password → JWT (httpOnly cookie)
Tier 2: Event editor       — edit token in URL → validated server-side, stored in session
Tier 3: Event attendee     — attendee token in URL → stored in session cookie
Tier 4: Anonymous visitor  — no token; cookie gives them a visitor_session only
                             (can still view public events, not RSVP without a token
                             if event is invite-only; can RSVP on open events)
```

Registered users exist across Tiers 3/4 — registration upgrades a cookie
session into a persistent account without changing what they can access.

---

## Tier 1 — Admin and Organizer

**Who:** the person(s) running the app. In a single-tenant deployment this is
typically one or two people.

**Auth flow:**
1. POST `/api/auth/login` with `{email, password}`.
2. Server verifies password (Argon2id), issues a signed JWT (1h TTL).
3. JWT is placed in an `httpOnly`, `SameSite=Strict`, `Secure` cookie named
   `auth_token`.
4. A `refresh_token` (7d TTL) is also set in a separate httpOnly cookie for
   silent renewal.
5. All admin/organizer API routes check the JWT via `authMiddleware`.

**Roles:**
- `admin` — full access: instance config, user management, all events.
- `organizer` — can create and manage their own events; cannot touch instance
  config or other users.

**First-run setup:**
If no `admin` user exists in the DB, any visit to the app redirects to
`/setup`. The setup form creates the first admin account. After that, only
admins can create additional organizer accounts.

---

## Tier 2 — Event Editor (Secret Edit Link)

**Who:** the event host and any co-hosts they share the link with.

**The token:**
- Generated at event creation: `crypto.randomBytes(32).toString('base64url')`.
- The **raw token** is returned once to the client and never stored again.
- The DB stores only `argon2id(token)` in `event_tokens.token_hash`.
- The edit URL looks like: `https://app.example.com/event/{slug}/edit/{token}`

**Auth flow:**
1. Request includes the raw token, either in the URL path or as an `X-Edit-Token`
   header for API calls.
2. Middleware fetches the `event_tokens` row where `type = edit` for this event,
   hashes the incoming token, and compares using `timingSafeEqual`.
3. On success, the raw token is stored in the visitor session for the duration
   of the session (so the host doesn't need to re-enter it on the same device).
4. The visitor_session record is flagged with `has_edit_access = true` for this
   event, so "My events" shows editor status correctly.

**Token delivery:**
- Shown once on screen after event creation with a "copy and save this link"
  warning.
- Emailed to the host if SMTP is configured and they provided an email.
- Stored in `localStorage` on the client as a secondary fallback (not a
  security boundary — just UX convenience).

**Recovery:**
- If lost: admin can regenerate the edit token from the admin dashboard, which
  invalidates the old one.
- In a future phase, a registered organizer can always access their events
  without the token (their JWT auth is sufficient).

---

## Tier 3 — Event Attendee Token

**Who:** guests who receive an invite link.

**The token:**
- Generated at event creation (or regenerated by the host): same mechanism
  as the edit token.
- A single attendee token can be shared with all guests (general invite link)
  or the host can generate multiple named tokens (e.g. "VIP", "general").
- Attendee token grants: view event details (if event is private), RSVP,
  comment, view guest list (if enabled).

**Auth flow:**
1. Guest follows the invite URL: `https://app.example.com/event/{slug}?t={token}`
2. Server validates token against `event_tokens` (type = attendee).
3. Token is stored in the visitor session cookie — subsequent visits to this
   event don't require the token in the URL.
4. The `visitor_sessions.event_access` JSONB column records `{slug: 'attendee'}`.

---

## Visitor Session — Cookie-Based Identity

This is the foundation for both Tier 3 and Tier 4. Every visitor gets one,
whether or not they hold a token.

**Cookie:**
- Name: `vsid` (visitor session ID)
- Type: httpOnly, SameSite=Lax, Secure
- Value: a signed, opaque session ID (not the DB UUID — map via a signature)
- TTL: 1 year, renewed on each visit

**What the session stores:**
- `visitor_sessions.id` — the stable identity
- `display_name` — learned from first RSVP, pre-fills future forms
- `email` — optional, collected at RSVP or when guest opts in for recovery
- `event_access` — JSONB map of `{slug → 'attendee' | 'editor'}` for quick
  rendering of "My events"
- `user_id` — null until the session is claimed by a registered account

**"My events" page:**
Reads `event_access` from the session's DB record and renders all linked events.
Shows past and upcoming events, with status (yes/maybe/no), and for past events
a link to the memories wall.

**Multi-device problem:**
A session lives in a browser cookie. A different device starts a fresh session.
Solutions (in priority order):
1. The original invite link still works — visiting it on a new device associates
   that device's session with the same event access.
2. If the guest provided an email, they can trigger a magic link from the
   "My events" page → clicking the link on the new device merges sessions.
3. Full account registration (Phase 2) — the durable solution.

---

## User Registration — Voluntary Upgrade (Phase 2)

Registration converts a cookie session into a permanent account. It is always
optional; never required to use the app.

**Incentives to register (shown in the UI):**
- "Your event history and memories across all devices"
- "Never lose your invite links"
- "Get notified about upcoming events"

**Registration flow:**
1. Guest clicks "Create account" from the "My events" page or an event page.
2. Form collects: email, password, display name (pre-filled from session).
3. POST `/api/auth/register` → creates `users` record with `role = member`.
4. Identity merge is triggered immediately (see below).
5. JWT + refresh token are issued; `vsid` cookie is updated to link to the
   new user.

**Login on a new device:**
1. POST `/api/auth/login` with email + password.
2. Server issues JWT; sets auth cookie.
3. The new device's visitor session is linked to the user (`visitor_sessions.user_id = user.id`).
4. All event access from the existing account is visible immediately.

---

## Identity Merge Flow

When a guest registers or logs in on a device that already has a visitor session,
we must merge the cookie session's history into the account without data loss.

```
Trigger: POST /api/auth/register or POST /api/auth/login
Input:   current visitor_session.id (from cookie) + newly authenticated user.id

Steps:
1. Load all RSVPs where rsvp.session_id = current_session.id
   → update each to set rsvp.user_id = user.id, rsvp.session_id = null
   (skip if an RSVP already exists for this user+event — keep the existing one)

2. Load all comments where comment.session_id = current_session.id
   → update each to set comment.user_id = user.id, comment.session_id = null

3. Load event_access map from current session
   → for each entry, upsert into user_event_access (Phase 2 table) or
     simply keep in session since the user is now logged in

4. Set visitor_sessions.user_id = user.id on the current session

5. If another session already exists for this user (prior device):
   → merge that session's event_access into the current session
   → mark the older session as superseded (keep for audit, not for active use)
```

This is atomic — wrap in a DB transaction. If it fails, roll back and return a
500; do not leave partially merged state.

---

## Stay-in-Touch Subscription (Phase 2)

Registered users can opt in to organizer broadcasts.

- `users.subscribed = true` means "send me news of upcoming events".
- Organizer can send a broadcast from the dashboard (subject + body).
- Delivery channels: email (always), Web Push (if user has granted permission),
  SMS (if configured and user provided a phone number).
- Unsubscribe link in every email; one-click in account settings.

This is explicitly NOT a social follow graph. There is one organizer per instance.
A user is either subscribed or not. The organizer does not need to "approve"
followers.

## Token Security Summary

| Token type   | Length   | Stored as      | Comparison         |
|--------------|----------|----------------|--------------------|
| Edit token   | 32 bytes | Argon2id hash  | timingSafeEqual    |
| Attendee token | 32 bytes | Argon2id hash | timingSafeEqual    |
| Session ID   | 32 bytes | Signed (HMAC)  | cookie verification|
| JWT          | —        | Signed (RS256) | jwt.verify()       |
| Magic link   | 32 bytes | SHA-256 hash   | timingSafeEqual    |
