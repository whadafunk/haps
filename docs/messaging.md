# Messaging Architecture

This document covers all outbound and inbound message flows in the platform:
host-to-guest updates (blasts), event invitations, system notifications, and
the longer-term evolution toward a guest inbox and peer-to-peer messaging.

---

## Concepts

**Update (blast)** — a host-authored message sent to all confirmed RSVPs for a
specific event. Already implemented as the Updates section on the manage page.
Stored as `event_messages` in the DB.

**Invitation** — a message that delivers an invite link to a prospective guest.
May be targeted (organiser picks specific contacts) or broadcast (open link).
Currently handled out-of-band (copy-paste the link); Phase 2 adds in-app
delivery.

**System notification** — generated by the platform on specific triggers:
RSVP reminder, waitlist promotion, event cancellation, etc.

**Direct message (Phase 5+)** — guest-to-guest or guest-to-host; not
planned until after mobile ships.

## Delivery Channels

### Email
- Cheapest at scale; works asynchronously; good for long content.
- Already in the Phase 1 plan (SMTP via Nodemailer).
- Required for invitations, reminders, blasts.
- Gracefully degrades: if SMTP is not configured, operations still succeed
  but no email is sent.

### Web Push
- Browser-based push notifications (Push API + Service Worker).
- No third-party service required; works on desktop and Android Chrome;
  iOS 16.4+ Safari supports it on installed PWAs.
- Requires opt-in prompt (shown on "My Events" page after first RSVP).
- Ideal for: real-time event updates while the guest is not on the page,
  invitation delivery, reminders.
- Zero marginal cost per message.

### In-app real-time (Server-Sent Events)
- When a blast or system event fires, guests currently viewing the event page
  get an immediate toast/banner without polling.
- Implemented with SSE (one endpoint per event, `text/event-stream`).
- No external infrastructure; handled entirely inside the Fastify process.
- Does not replace persistent channels — if the guest isn't on the page,
  they won't see it until they return (which is why SSE + Web Push are
  complementary).

### SMS
- High deliverability; works on feature phones and anywhere with signal.
- Most expensive per message (~$0.01–0.05/segment depending on country and
  carrier, via Twilio or equivalent).
- No formatting; 160-char segments.
- Good fit for: last-minute reminders, critical cancellation notices.
- Only enabled when `TWILIO_*` env vars are set.

### WhatsApp (Business API)
- Lower cost per conversation than SMS ($0.005–0.015) and richer formatting.
- Very high penetration in Europe, LatAm, Middle East, Africa.
- **Constraint**: all outbound messages to users who have not messaged the
  business in the last 24h must use pre-approved Meta message templates.
  Arbitrary free-text blasts are not allowed outside the 24h service window.
  Templates must be submitted to Meta for approval before use.
- Requires a Meta Business Account, WhatsApp Business API access (via a
  BSP such as Twilio, MessageBird, or direct Meta Cloud API), and a
  dedicated phone number.
- Suitable for: invitation delivery, reminders, cancellation notices
  (all template-able).
- Not suitable for: ad-hoc freeform blasts (unless the guest initiated
  the conversation recently).
- Only enabled when `WHATSAPP_*` env vars are set.

### Mobile Push (APNs / FCM) — Phase 4
- Native push for the React Native / Expo mobile app.
- Replaces Web Push for users who have the mobile app installed.
- Managed via Expo's push notification service (abstracts APNs/FCM).

## Guest Inbox

A feed of all notifications across events the guest is connected to. Natural
evolution of "My Events" — instead of just a list of events, the page shows
a chronological timeline of actions that require attention or are of interest.

### What appears in the inbox
| Item | Trigger |
|------|---------|
| New invitation | Host sends an invite to the guest's email/phone |
| Event update | Host publishes a blast on an event the guest RSVPed to |
| RSVP reminder | 7 days, 2 days, 1 day before event start |
| Waitlist promotion | Guest moved from waitlist to confirmed |
| Event cancellation | Host cancels an event the guest RSVPed to |
| Event date change | Host updates `starts_at` on a published event |

### DB schema (Phase 2)

```sql
notifications
  id            uuid        primary key default gen_random_uuid()
  session_id    uuid        references visitor_sessions(id) on delete cascade
  user_id       uuid        references users(id) on delete cascade
  -- one of session_id or user_id is set, not both
  event_id      uuid        references events(id) on delete cascade
  type          text        not null
                -- 'invitation' | 'event_update' | 'reminder' | 'waitlist_promotion'
                -- | 'event_cancelled' | 'event_changed'
  body          text                             -- short human-readable summary
  link          text                             -- deep link (event page or specific section)
  read          boolean     not null default false
  created_at    timestamptz not null default now()
```

### UI
- Badge on the "My Events" nav item showing unread count.
- Inbox tab on the "My Events" page (or a dedicated `/inbox` route).
- Clicking a notification marks it read and navigates to the relevant link.
- "Mark all read" action.

### No real-time required
The inbox is populated on write (platform inserts a row when the triggering
action happens). On next page visit the guest sees the unread items.
Real-time delivery of new inbox items is handled by Web Push or in-app SSE
for active sessions — the inbox row is the persistent record; the push/SSE
is the ephemeral signal.

## In-App Real-Time Notifications (SSE)

When a blast or system notification is created, connected clients watching
that event receive an SSE event immediately.

```
GET /api/events/:slug/stream
  Auth:    attendee token in session (or open event)
  Stream:  text/event-stream
  Events:
    event: update
    data: { type: 'blast', id, subject, body, createdAt }

    event: update
    data: { type: 'rsvp_count', yesCount, maybeCount }

    event: update
    data: { type: 'status_change', status: 'cancelled' }
```

Client subscribes via `new EventSource('/api/events/:slug/stream')` when the
event page mounts. On receiving an update:
- `blast` → insert the new update card at the top of the Updates list, show a
  toast "New update from the host".
- `rsvp_count` → update the guest count badge without a full reload.
- `status_change` → banner "This event has been cancelled."

Fastify manages open connections in a per-event Set. Connections are closed
after 2h (or on client disconnect) to avoid resource leaks.

## Invitation Flow (Phase 2)

Organiser composes an invitation from the event manage page:

1. Selects contacts from address book (email and/or phone).
2. Writes an optional personal message (defaults to event title + date).
3. System generates a unique attendee token per contact (if `invite_only`
   event) or uses the general token (if open event).
4. Delivers via all enabled channels for that contact (email, SMS, WhatsApp).
5. Creates a `notifications` row for contacts who are known guests
   (have an existing session or account).
6. Delivery status tracked per invitation per channel.

```sql
invitations
  id            uuid        primary key default gen_random_uuid()
  event_id      uuid        not null references events(id) on delete cascade
  token_id      uuid        references event_tokens(id)  -- the per-invite token
  recipient_email  text
  recipient_phone  text
  personal_message text
  sent_at       timestamptz
  created_at    timestamptz not null default now()

invitation_deliveries
  id            uuid        primary key default gen_random_uuid()
  invitation_id uuid        not null references invitations(id) on delete cascade
  channel       text        not null    -- 'email' | 'sms' | 'whatsapp' | 'push'
  status        text        not null    -- 'queued' | 'sent' | 'delivered' | 'failed' | 'bounced'
  external_id   text                    -- provider message ID for webhook reconciliation
  updated_at    timestamptz not null default now()
```

## Channel Priority and Fallback

When sending a notification, channels are tried in priority order. A guest
who has opted in to push and provided an email gets both; a guest with only
an email gets email only.

```
Web Push (if subscribed + browser supports it)
  ↓ also
Email (if email on record)
  ↓ also
WhatsApp (if phone on record + WhatsApp configured)
  ↓ fallback if WhatsApp fails
SMS (if Twilio configured)
```

"Also" means: send on all available opted-in channels, not stop at the first
success. The guest decides which channels to receive on in their preferences
(Phase 2). In Phase 1, email is the only channel; all others add incrementally.

## Future: Peer-to-Peer Messaging (Phase 5+)

Once the mobile app ships and the guest inbox exists, direct messaging becomes
feasible. Likely shape:

- Guest-to-host DM (question about the event, dietary note, etc.)
- Host replies in thread
- Guest-to-guest limited to event context (not a general chat)
- All messages stored in a `direct_messages` table; delivered via Web Push /
  mobile push + inbox

This is out of scope until Phase 4 is complete. The infrastructure decisions
(SSE endpoint, notifications table, push subscription management) made in
Phases 2–3 are designed to extend naturally to DMs without a rewrite.
