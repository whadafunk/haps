# API Conventions (Fastify + TypeScript)

## File Structure

```
packages/api/src/
  routes/
    events.ts       CRUD + token management
    rsvps.ts
    comments.ts
    auth.ts
    session.ts
    admin.ts
  middleware/
    auth.ts         JWT verification → sets request.user
    editToken.ts    Edit token verification → sets request.isEditor
    session.ts      Visitor session cookie → sets request.session
    rateLimit.ts    Per-route rate limit config
    errorHandler.ts Fastify setErrorHandler
  db/
    schema.ts       Drizzle table definitions (single source of truth)
    migrations/     Auto-generated by `drizzle-kit generate`
    index.ts        DB connection singleton
  services/
    events.ts       Business logic (no raw SQL here)
    tokens.ts       Token generation, hashing, validation
    email.ts        Nodemailer wrapper
    storage.ts      File upload abstraction (local / S3)
  lib/
    crypto.ts       Token gen, argon2id helpers, timingSafeEqual wrapper
    config.ts       Typed env var parsing (with Zod)
  app.ts            Fastify instance creation + plugin registration
  server.ts         Entry point (listen)
```

## Route Pattern

Every route file exports a Fastify plugin. Use the async plugin pattern:

```typescript
import { FastifyPluginAsync } from 'fastify'
import { z } from 'zod'
import { db } from '../db/index.js'

const eventsRoute: FastifyPluginAsync = async (fastify) => {
  fastify.post('/events', {
    preHandler: [fastify.authenticate],  // middleware hooks by name
    schema: {
      body: zodToJsonSchema(CreateEventSchema),  // use fastify-zod or manual conversion
    },
  }, async (request, reply) => {
    const body = CreateEventSchema.parse(request.body)
    // ...
    return reply.code(201).send({ event, editToken, editLink })
  })
}

export default eventsRoute
```

## Error Handling

Throw typed errors; the central error handler formats them:

```typescript
// Throw this anywhere in a route or service:
throw fastify.httpErrors.notFound('EVENT_NOT_FOUND')
// or for custom codes:
import { createError } from '../lib/errors.js'
throw createError(403, 'INVALID_EDIT_TOKEN', 'Edit token is invalid or expired.')
```

The `errorHandler` maps these to:
```json
{ "error": { "code": "INVALID_EDIT_TOKEN", "message": "...", "details": {} } }
```

Never leak stack traces or DB error messages to the client. Log them server-side.

## Validation

- All request bodies, query strings, and URL params go through a Zod schema.
- Use `.strict()` on body schemas to reject unknown fields.
- Define schemas in `packages/shared/src/schemas/` so the web package can
  reuse them for client-side form validation.
- Coerce dates: `z.string().datetime()` for ISO strings; convert to `Date`
  in the service layer, not in the route handler.

## DB Access

- Raw DB queries go in `services/` only. Route handlers call services.
- Use Drizzle's query builder. Avoid raw SQL except for complex aggregations
  where it's significantly clearer.
- Transactions: use `db.transaction(async (tx) => { ... })`. Pass `tx` to
  service functions that must run within the transaction.
- Never select `*`; always specify columns explicitly.

## Auth Middleware

Three decorators available on the Fastify instance:

```typescript
fastify.authenticate          // verifies auth_token JWT; throws 401 if missing/invalid
fastify.requireRole('admin')  // verifies JWT + checks role
fastify.requireEditToken      // verifies X-Edit-Token or session-stored token
fastify.loadSession           // always succeeds; populates request.session from vsid cookie
```

`loadSession` runs on every request (registered globally). `authenticate` is
added per-route where needed.

## Logging

Use Fastify's built-in Pino logger. Never log:
- Raw tokens (edit, attendee, session IDs)
- Passwords or password hashes
- Email addresses at debug level (only at info+ with explicit justification)

Log format: JSON in production, pretty-print in development (via `pino-pretty`).
